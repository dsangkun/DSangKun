## 一、当前目标

当前只推进 **Amazon运营推进器 - 模块3：运营数据展示** 的数据库方向，不扩展到订单、库存、采购、权限等其他模块。

目标部署环境：

- 阿里云服务器
- MySQL 5.7.44

因此本次表结构、字段、索引、约束全部按 **MySQL 5.7 兼容** 设计。

---

## 二、当前业务口径（已经固定）

### 1）前端主展示对象

当前前端页面中：

- **一个卡片 = 一个子 ASIN = 一个具体产品**
- 父 ASIN 不作为前端卡片直接展示对象
- 父 ASIN 仅用于：
  - 汇总
  - 归属关系
  - 对账校验
  - 统计支撑

### 2）店铺口径

当前业务处理链中：

- **非 SBV 数据按店铺下载与导入**
- 原始数据层必须保留店铺字段
- 但前端展示层 **不体现店铺概念**

因此数据库采用：

- 原始层保留 `shop_name`
- 展示层按子 ASIN 归集
- 前端接口默认不把店铺作为主筛选维度暴露

## 三、数据分层

本次数据库设计分两层。

### 1）原始导入层

用途：保留 Excel / CSV / 报表的原始结构，便于追溯、重算、纠错。

表：

- `m3_import_batch`
- `m3_import_file`
- `m3_biz_report_parent_daily`
- `m3_biz_report_child_daily`
- `m3_sbv_campaign_daily`
- `m3_sp_campaign_daily`
- `m3_review_daily`

### 2）展示汇总层

用途：直接服务前端产品卡片展示。

表：

- `m3_product`
- `m3_product_asin_map`
- `m3_product_daily_summary`

## 四、各表职责

### 1. `m3_product`

产品主表。

用途：

- 存前端卡片对应的产品基础资料
- 当前主展示对象是 **子 ASIN 产品**
- 支撑前端卡片的名称、图片、市场、主子 ASIN 等展示

### 2. `m3_product_asin_map`

产品与 ASIN 的映射关系表。

用途：

- 建立父 ASIN / 子 ASIN / 广告 ASIN 与产品卡片之间的关系
- 支撑业务报表、广告报表向产品维度归集
- 兼容后续一个产品绑定多个 ASIN 的情况
- 兼容 **一个父 ASIN 对多个子 ASIN 产品** 的正常业务场景，因此唯一约束应落在 `product_id + asin + asin_role` 组合上，而不是把 `asin` 单独做全局唯一

### 3. `m3_import_batch`

导入批次表。

用途：

- 记录每次导入的批次级信息
- 保存导入目录、日期、状态、执行人

### 4. `m3_import_file`

导入文件登记表。

用途：

- 记录每个导入文件
- 保存文件类型、路径、MD5、解析状态、错误信息

### 5. `m3_review_daily`

产品评价日报。

用途：

- 存评分、评价数、新增评价数、问答数等
- 服务前端评价区块

### 6. `m3_biz_report_parent_daily`

父 ASIN 业务日报原始表。

用途：

- 存父 ASIN 粒度的流量、销售、转化数据
- 主要用于对账和父级汇总校验

### 7. `m3_biz_report_child_daily`

子 ASIN 业务日报原始表。

用途：

- 存子 ASIN 粒度的流量、销售、转化数据
- 是前端卡片销售/流量数据的核心来源之一

### 8. `m3_sbv_campaign_daily`

SBV 广告活动日报原始表。

用途：

- 记录 SBV 广告活动明细
- 支撑前端广告模块展开明细表
- 保留 `is_summary_row`、`acos_raw`、`cvr_raw` 兼容异常值和汇总行

### 9. `m3_sp_campaign_daily`

SP 广告活动日报原始表。

用途：

- 记录 SP 广告活动明细
- 支撑前端广告模块展开明细表

### 10. `m3_product_daily_summary`

产品日报汇总表。

用途：

- 直接给前端产品卡片读取
- 当前按 **子 ASIN 产品维度** 汇总
- 汇总评价、销售、流量、广告关键指标

## 五、已确认的数据来源链路

当前真实数据来源链路包括：

- 流量数据统计 ASIN 映射表
- N 张单产品 SBV 表
- SBV 汇总表（人工中间结果）
- 业务报告父 ASIN
- 业务报告子 ASIN
- SP 广告报告
- 最终统计报表（对账/验收）
- 案例数据（前端广告模块验收样本）

已确认样本中的关键文件角色：

1. 映射表：建立父/子 ASIN 关系及是否跑 SBV 标记
2. 子业务报告：子 ASIN 卡片销售/流量主数据源
3. 父业务报告：父级对账与汇总校验数据源
4. SP 广告报告：SP 广告汇总与活动明细数据源
5. SBV 单产品报表：SBV 活动明细数据源
6. 最终统计报表：汇总结果对账源
7. 案例数据：前端广告模块展示验收源

## 六、字段映射原则

### 1）产品主对象

- `m3_product.product_code`：先直接用子 ASIN
- `m3_product.product_name`：优先取映射表中的子 ASIN 产品名，缺失时回落到子业务报告标题
- `m3_product.primary_asin`：当前主展示子 ASIN

### 2）销售 / 流量

子业务报告中的以下字段进入 `m3_biz_report_child_daily`，并进一步汇总到 `m3_product_daily_summary`：

- `会话数 - 总计` -> `sessions_total`
- `页面浏览量 - 总计` -> `page_views_total`
- `已订购商品数量` -> `units_ordered`
- `订单商品总数` -> `order_item_count`
- `已订购商品销售额` -> `sales_amount`
- `商品会话百分比` -> `unit_session_pct`
- `转化率 - 总计` -> `conversion_rate_total`
- `推荐优惠/推荐展示位百分比` -> `buy_box_pct`

### 3）广告

SP 与 SBV 在原始层分表保存，在展示层统一标准化输出。

统一广告展示字段：

- `campaign_name`
- `impressions`
- `clicks`
- `ctr`
- `cpc`
- `spend`
- `ad_sales`
- `acos`
- `ad_orders`
- `cvr`

其中：

- SP 来源于 `m3_sp_campaign_daily`
- SBV 来源于 `m3_sbv_campaign_daily`
- 产品卡片汇总结果进入 `m3_product_daily_summary`

## 七、清洗规则

### 数值类

1. 去除千分位逗号：`1,548 -> 1548`
2. 去除货币符号：`US$5,339.29 -> 5339.29`
3. 空值、`--`、空字符串 -> `0` 或 `NULL`
4. 百分比字符串：`12.81% -> 0.1281`
5. 普通数字文本直接转 decimal/int

### 异常文本类

SBV 中存在异常值，例如：

- `ACoS = 有花费无销售额`
- `CPA = 有花费无订单`

处理策略：

- 原始异常文本保留在 `*_raw` 字段
- 数值字段可记为 `NULL`
- 汇总计算时按明确规则处理，避免 silently 变成错误 0 值

### 结构类

- SBV 第 0 行通常是汇总行，导入活动明细时应标记 `is_summary_row=1`
- 映射表父级字段需要做向下填充，兼容 Excel 合并单元格
- 同一天同一子 ASIN 可能来自不同源文件，但最终展示按产品维度汇总

## 八、为什么这版比旧脚本更适合作为当前基线

相比 `02` / `03` 两个早期脚本，这版的提升主要有：

1. **明确了前端主维度就是子 ASIN 产品**，不再停留在模糊宽表阶段
2. **把原始层和展示层拆开**，后续 ETL 和 API 更清晰
3. **把 SBV / SP 的活动明细表结构固定下来**，能直接支撑前端广告明细展开
4. **补齐导入批次 / 文件追溯能力**，便于后续做重算、排错、补数
5. **完全按 MySQL 5.7 兼容设计**，可直接用于阿里云部署
