# 2026-04-29 钉盘接口对接阻塞说明

> 目的：记录本轮钉钉钉盘（Drive）联调中已经验证过的事实、当前真实阻塞点、以及继续向钉钉支持追问时应使用的固定口径。

---

## 一、当前已确认的事实

### 1. 基础鉴权链路是通的

已确认当前 `appKey` / `appSecret` 可以成功换取：
- `access_token`

这说明：
- 应用凭证本身有效
- 不是基础鉴权失败
- 当前问题不在 token 获取

### 2. 新版钉盘空间接口路径已确认

已确认空间列表接口使用：
- `GET /v1.0/drive/spaces`

请求头使用：
- `x-acs-dingtalk-access-token: {access_token}`

不是：
- `Authorization: Bearer ...`

### 3. 空间列表接口已进入更深一层校验

联调结果表明：
- 调空间列表时，已经不再报最基础的权限缺失
- 现阶段会进入 `unionId` 相关参数/身份校验

这至少说明：
- 当前应用对 Drive 空间体系不是完全不可达
- `Drive.Space.Read` 至少已部分打通或接口可被网关识别

### 4. 文件列表接口的真实报错很明确

当前调用的文件列表接口为：
- `GET /v1.0/drive/spaces/{spaceId}/files?parentId=0&maxResults=...`

返回的关键结论是：
- 网关明确要求 `Drive.File.Read`

这说明：
- 不是路径拼错导致的 404
- 不是 token 完全无效
- 不是 `Storage.File.Read` 之类权限可以直接覆盖的情况
- 当前阻塞点就是：**钉钉平台对 `/v1.0/drive/spaces/{spaceId}/files`` 这条能力要求 `Drive.File.Read`，但控制台无法申请该权限**

---

## 二、已经验证过、可以停止重复试错的点

### 1. 控制台内搜索不到 `Drive.File.Read`

已多轮验证：
- 当前应用权限申请界面中找不到 `Drive.File.Read`
- 即使通过官方 deep link 跳转申请页，也看不到该权限项
- 新建应用后仍然看不到该权限项

结论：
- 这不是单个应用配置偶发异常
- 很可能是权限未开放、已下线、或仅对白名单/特定应用类型开放

### 2. `Storage.File.Read` / `Storage.DownloadInfo.Read` 不能直接替代

已验证：
- 即使参考 doc / storage 体系文档和元数据，drive 文件列表接口网关仍然明确要求 `Drive.File.Read`
- `Storage.File.Read` 对应的是另一套文档/存储接口能力，不等于 `/v1.0/drive/spaces/{spaceId}/files`

结论：
- 不能把 storage 权限当作 drive 文件列表权限的替代品
- 不能在未拿到官方明确映射前，贸然把现有实现切去另一套接口体系

### 3. 当前不应继续盲猜权限名

已尝试或已排除方向包括：
- 继续搜索 `Drive.File.Read`
- 尝试用 storage 权限代替
- 根据旧认知混用 Bearer / 旧路径 / 新路径

结论：
- 现阶段再继续盲试，收益很低
- 真正缺的是**钉钉官方给出明确且可落地的接口-权限-应用类型映射**

---

## 三、当前技术结论

### 结论 1：代码方向本身没有明显跑偏

后端当前已对齐到较正确的 drive 调用骨架：
- 使用 `x-acs-dingtalk-access-token`
- 使用真实 `/v1.0/drive/...` 路径
- 能正常拿到远端错误并透传

因此当前主要问题不是本地代码骨架，而是：
- 平台权限开放策略
- 文档、控制台、网关三者不一致

### 结论 2：主阻塞不在实现，而在平台能力可申请性

当前最核心阻塞是：
- 文件列表网关要求 `Drive.File.Read`
- 但开放平台控制台找不到 `Drive.File.Read`
- 官方支持曾建议改走 `TopAPI /topapi/file/get_file_list` 或 `cspace` 体系
- 但截至当前，尚未从公开 OpenAPI 元数据中核实到该建议对应的：
  - 准确官方文档链接
  - 准确接口路径与方法
  - 准确权限码
  - 适用应用类型

因此现在不能贸然切换实现。

## 四、当前系统内已做的保护性处理

为避免首页或调试页被钉盘权限阻塞拖垮，当前后端建议按以下思路维护：

1. 保留钉钉调试入口，用于单独探测：
   - token 是否获取成功
   - 空间列表是否可访问
   - 文件列表是否因缺少 `Drive.File.Read` 被拒绝

2. 首页聚合逻辑对钉盘文件列表异常做降级：
   - 即使 `Drive.File.Read` 未开放
   - 概览接口也不应整体报错
   - 当前可先按 `0` 降级，等待官方给出明确替代接口

这类处理的目标是：
- 保留联调证据
- 不让暂时不可控的平台权限问题拖垮主页面

## 五、下一步最优动作

### 优先级最高：继续追问钉钉支持

不要继续盲试权限名。

下一步应继续要求钉钉支持明确提供以下信息：

1. **如果 `/v1.0/drive/spaces/{spaceId}/files` 不是普通应用当前可用方案，请给出官方推荐替代接口的准确文档链接**
2. **请给出准确请求路径与 HTTP 方法**
3. **请给出该接口所需的准确权限码（scope）**
4. **请说明适用的应用类型（企业内部应用 / 第三方企业应用 / 个人应用 / 管理侧接口等）**
5. **请明确 `Drive.File.Read` 当前是否开放申请；如果不开放，请说明开放范围或替代方案**

### 次优动作：拿到支持回复后先核验，再改代码

如果支持给出替代接口，先核验：
- 官方文档是否真实存在
- 接口路径/方法是否真实可调用
- 权限码是否能在控制台申请
- 适用应用类型是否与当前项目一致

只有这四项都成立，再进入代码适配。

## 七、对“方案一：切换到 TopAPI / 团队空间接口”的阶段性评估

收到的建议大意是：
- 改用 `POST /topapi/file/get_file_list`
- 在控制台搜索“钉盘”/`cspace`
- 可能需要类似 `cspace:file:read` 的权限

### 当前评估结论

**结论：方向上“可能有机会”，但截至当前证据强度，仍不能直接判定为“已确认可行并可立即切换实现”。**

原因如下：

1. 该建议目前仍然停留在“关键词级别”
   - 只给了 `TopAPI` / `cspace` / `可能为 cspace:file:read 或类似表述`
   - 没有给出可核验的官方文档链接
   - 没有给出准确权限码名称
   - 没有给出适用应用类型

2. 当前我们已经有一条明确被网关验证过的事实链
   - `GET /v1.0/drive/spaces/{spaceId}/files` 明确要求 `Drive.File.Read`
   - 这条事实来自真实调用返回，不是猜测
   - 因此任何替代方案都必须拿出同等级别的官方证据，才能替换现有实现方向

3. 本轮基于本地已抓取的公开 OpenAPI / 文档线索再次核验后，尚未拿到足够硬的公开证据，能直接证明：
   - `POST /topapi/file/get_file_list` 就是当前企业内部应用的官方推荐稳定替代接口
   - 其所需权限码准确就是 `cspace:file:read`
   - 该权限在当前应用类型下确实可申请

### 因此当前判断应为

- **不是直接否定该方案**
- 但也**不能因为它“听起来像对的”就直接切主实现**

更准确的说法是：

> `TopAPI /topapi/file/get_file_list` 这条路“有待进一步核验”，目前证据不足，只能列为候选替代方案，不能列为已确认可实施方案。

### 什么情况下，才可以判定该方案可行

至少需要同时满足以下 4 条：

1. 钉钉支持给出**准确官方文档链接**
2. 文档中能对应到**准确接口路径 / 请求方法 / 参数说明**
3. 文档或控制台能对应到**准确权限码名称**
4. 能确认该接口**适用于当前应用类型**，且权限在控制台中确实可申请

在这 4 条没齐之前，不建议把后端主实现从当前 `/v1.0/drive/...` 骨架迁走。

### 下一步最合理动作

继续追问钉钉支持，要求其补齐以下信息：

- `POST /topapi/file/get_file_list` 的官方文档链接
- 准确权限码（不要“类似于”）
- 控制台申请入口截图或准确权限名称
- 适用应用类型
- 与当前 `/v1.0/drive/spaces/{spaceId}/files` 的关系说明
