# 模块3：运营数据数据库部署与导入方案

更新时间：2026-04-02

## 1. 关键修正

此前把 `流量数据统计报告.xlsx` 当成主源数据来设计导入，这个认知现在需要修正。

你刚补充的结构是对的：

**每日原始数据由 5 张表组成，最终“流量数据统计报告”是汇总结果，不是原始源。**

这 5 张表分别是：

1. `SBV数据/<日期>/汇总.xlsx`
2. `店铺目录/<日期>/业务报告-父ASIN.csv`
3. `店铺目录/<日期>/业务报告-子ASIN.csv`
4. `店铺目录/<日期>/SP广告报告-<日期>.xlsx`
5. `流量数据统计ASIN映射表.xlsx`

因此现在数据库设计必须改成：

**原始五表入库 -> 汇总口径重建 -> 后端接口聚合 -> 前端展示**

而不是：

**最终汇总报表入库 -> 直接给前端用**

## 2. 现在应该从哪一步开始

### 当前第一步
**先在服务器 MySQL 建“原始数据层”表。**

已新增脚本：
- `database/03-operation-raw-init.sql`

这个脚本按五张原始表拆分了入库表：

- `op_source_file`
- `op_asin_mapping`
- `op_business_parent_daily`
- `op_business_child_daily`
- `op_sp_ad_daily`
- `op_sbv_daily`

## 3. 为什么要这么拆

因为五张原始表的粒度不同：

- 父 ASIN 业务报告：父体维度
- 子 ASIN 业务报告：子体维度
- SP 广告报告：广告明细维度
- SBV 汇总：子体广告汇总维度
- ASIN 映射表：父子关系与产品命名维度

如果现在强行塞进一张表，后面统计口径一定会乱。

## 4. 推荐落地顺序

### 第 1 步：服务器执行建库脚本
执行：
- `database/03-operation-raw-init.sql`

### 第 2 步：先导入这 5 张原始表
建议先只做 `EXPERLAM-美国`，不要一上来连 `KHINYA` 一起导。

优先顺序：

1. 导 `流量数据统计ASIN映射表.xlsx`
2. 导 `业务报告-父ASIN.csv`
3. 导 `业务报告-子ASIN.csv`
4. 导 `SP广告报告.xlsx`
5. 导 `SBV数据/<日期>/汇总.xlsx`

### 第 3 步：做日汇总重建逻辑
也就是把上述五张原始表，重新汇成你现在看到的“流量数据统计报告”口径。

### 第 4 步：再给后端接口用
等汇总口径稳定后，再改：
- `GET /api/workbench/operation-data`

## 5. 当前最稳妥的工程策略

### 原始层
保留完整来源字段，能追溯、能重算。

### 汇总层
后续单独建：
- 父体运营日汇总
- 子体运营日汇总
- 模块3接口用聚合表/视图

### 展示层
前端模块3只查聚合层，不直接查原始层。

## 6. 现在的直接结论

**现在应该先从“原始五表入库”开始，不再以最终流量统计报表作为主导入对象。**

如果继续开发，下一步最应该做的是：

1. 在服务器执行 `database/03-operation-raw-init.sql`
2. 我再继续帮你写对应的原始五表导入脚本
3. 导入 `EXPERLAM-美国` 最近几天数据做验证
4. 验证通过后再批量导全量历史数据
